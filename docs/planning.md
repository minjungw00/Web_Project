# 1. 서비스 철학 및 목적

## 1.1 서비스 목적

- 개발 역량을 구조적으로 정리하고 공개한다.
- 학습 내용을 장기적으로 축적 가능한 아카이브 형태로 관리한다.
- 기술 이해도를 결과물이 아닌 사고 과정 중심으로 증명한다.

## 1.2 서비스 성격 정의

이 서비스는 단순 블로그가 아니라 다음 성격을 동시에 가진다:

- 포트폴리오 전시 공간
- 기술 사고 기록 공간
- CS 개념 정리 지식 베이스
- 실험적 구현 결과물 허브

# 5. Portfolio

## 5.1 목적

- 프로젝트 결과물 증명
- 설계 사고 과정 전달
- 문제 해결 경험 구조화

## 5.2 콘텐츠 구조

### 목록 페이지 목적

- 관심 유도
- 핵심 기술 키워드 노출
- 프로젝트 차별성 강조

### 프로젝트 카드 구성 요소

- 썸네일 (핵심 기능 화면)
- 프로젝트명
- 프로젝트 유형 (개인/팀 + 역할)
- 진행 기간
- 핵심 기술 (2~4개)
- 짧은 설명
- 핵심 가치 한 줄 요약

### 상세 페이지 콘텐츠 구조

1. 프로젝트 개요
   - 한 문단 요약
   - 프로젝트 동기
2. 문제 정의
   - 해결하려던 문제
   - 배경
   - 목표
3. 설계 및 구조 설명
   - 아키텍처 개요
   - 주요 설계 결정
   - 데이터 흐름 설명
4. 핵심 경험 (3~4개)
   - 각 항목은 다음 구조를 따른다:
     - 문제 상황
     - 고려한 선택지
     - 판단 기준
     - 최종 선택 이유
     - 적용 결과 (정량 지표 포함 가능)
     - 배운 점
5. 성과 및 결과
   - 개선 수치
   - 운영 경험
   - 배포 결과
6. 회고 및 개선점
   - 한계
   - 확장 가능성

## 5.3 시각 전략

- 블록 단위 구획
- 문제-해결 단위 시각적 분리
- 스캔 친화적 구조

## 5.4 이미지 사용 원칙

허용 목적:

- 아키텍처 설명
- 성능 비교
- 요청 흐름
- 핵심 기능 설명

지양:

- 단순 UI 나열
- 설명 없는 이미지
- 동일 유형 반복 이미지

# 7. CS Docs

## 7.1 목적

CS Docs는 단순 개념 문서 나열형 구조가 아니라,  
**관계 기반 지식 구조를 시각적으로 탐색하는 학습 공간**을 지향한다.

목표는 다음과 같다:

- 개념을 고립된 텍스트로 소비하지 않도록 한다.
- 선후 관계와 연관 구조를 직관적으로 이해하도록 돕는다.
- 암기 중심 학습이 아니라 구조 이해 중심 학습을 유도한다.
- 면접 대비 시 개념 간 연결 사고를 강화한다.

## 7.2 지식 구조 모델: 뉴런–시냅스 메타포

CS Docs는 **뉴런-시냅스 스타일의 지식 그래프 구조**를 채택한다.

### 구조 개념

- 허브 노드 → 대분류 개념 (세포체)
- 가지 노드 → 하위 개념 (수상돌기)
- 교차 연결 → 개념 간 관계 (시냅스)

이 구조는 단순 카테고리 트리와 다르다.

- 하나의 개념은 여러 상위 개념과 연결될 수 있다.
- 선형 목차 구조가 아닌 네트워크 구조를 가진다.
- 학습은 “위에서 아래로”가 아니라 “관계 중심 확장” 형태로 진행된다.

## 7.3 사용자 상호작용 경험

사용자는 그래프 화면에서 다음 경험을 한다:

1. 대분류 허브(예: OS, DB, Network 등)를 중심으로 전체 구조를 본다.
2. 특정 개념 노드를 선택한다.
3. 해당 개념의 문서가 열리며 아래 내용을 확인한다.
   - 정의
   - 핵심 원리
   - 예시
   - 선행 지식
   - 연관 개념
   - 면접 질문
4. 문서 내부에서 선행/연관 개념으로 이동한다.
5. 그래프 상에서 개념 간 연결을 시각적으로 확인한다.

이 과정을 통해 사용자는:

- “이 개념이 왜 중요한지”
- “어떤 개념 위에 서 있는지”
- “어디에 사용되는지”

를 동시에 이해하게 된다.

## 7.4 개념 관계 정의 원칙

그래프는 임의 연결이 아니라, 명확한 관계 유형을 가진다.

### 관계 유형

- PREREQ  
   → 이해를 위해 반드시 필요한 선행 개념
- RELATED  
   → 직접적 선후 관계는 아니지만 밀접히 연결된 개념
- PART_OF  
   → 상위 구조의 구성 요소
- CONTRAST  
   → 비교 학습이 필요한 대조 개념

각 관계는 학습 흐름을 의도적으로 설계한다.

## 7.5 개념 문서 템플릿

각 개념 문서는 다음 구조를 따른다:

1. 한 줄 요약
2. 정의
3. 핵심 원리
4. 예시 또는 코드
5. 선행 지식
6. 연관 개념
7. 면접 질문 예시

문서는 독립적으로 읽히지만,  
항상 다른 개념으로 확장 가능해야 한다.

## 7.6 학습 경험 목표

CS Docs는 다음을 목표로 한다:

- 개념을 트리 구조가 아닌 네트워크 구조로 이해하게 한다.
- “이 개념은 무엇인가?”에서 멈추지 않고  
   “이 개념은 어디에 연결되는가?”까지 생각하게 만든다.
- 실무 적용과 면접 질문을 동시에 대비할 수 있도록 구성한다.
- 장기적으로 확장 가능한 지식 베이스를 형성한다.

## 7.7 차별성

일반 블로그의 CS 정리와 달리:

- 글 목록 나열 구조가 아니다.
- 카테고리 기반 단순 분류가 아니다.
- 개념 간 관계가 시각적으로 드러난다.
- 탐색이 학습의 일부가 된다.

CS Docs는 “읽는 공간”이 아니라  
**탐색하며 이해하는 공간**이다.

# 10. 콘텐츠 간 연결 전략

- Portfolio ↔ Blog: 프로젝트 상세 구현 분석 연결
- CS Docs ↔ Blog: 개념 심화 글 연결
- Mini Games ↔ Portfolio: 구현 경험 설명 연결
- CS Docs ↔ Portfolio: 사용된 이론 개념 연결

서비스는 독립된 섹션의 집합이 아니라  
**상호 연결된 지식 네트워크 구조**를 지향한다.
